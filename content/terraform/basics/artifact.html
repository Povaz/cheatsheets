01 What Terraform is (and why)

Terraform is an Infrastructure as Code (IaC) tool: you describe the infrastructure you want — servers, databases, networks, buckets — in text files, and Terraform makes the real world match. HashiCorp's own one-liner: "Terraform is an infrastructure as code tool that lets you build, change, and version cloud and on-prem resources safely and efficiently."

The shift in mindset: instead of clicking around a cloud console or running ad-hoc CLI commands, you write a desired-state description and commit it to git. The benefits that follow:

  • Declarative — you state what you want, not the step-by-step how. Terraform figures out the steps.
  • Versioned & reviewable — infrastructure changes go through pull requests like any other code.
  • Reproducible — the same configuration produces the same infrastructure, in dev, staging, and prod.
  • Multi-platform — one tool and one language across AWS, Azure, GCP, Kubernetes, GitHub, Datadog, and hundreds more.

Source: What is Terraform — HashiCorp Developer

02 The mental model: Write → Plan → Apply

Before any syntax, internalise the loop you'll repeat forever. HashiCorp calls it the core workflow, and it has exactly three steps:

1 · Write

Author your infrastructure as code in .tf files.

→

2 · Plan

Preview the changes Terraform would make — before anything happens.

→

3 · Apply

Approve, and Terraform provisions the real resources.

The Plan step is what makes Terraform safe to use: it always shows you exactly what will be created, changed, or destroyed, and waits for your approval before touching anything. You'll see this loop in action in §7.

Source: Core Terraform Workflow — HashiCorp Developer

03 Installing Terraform

Terraform is a single binary — install it, and you have the terraform command. On macOS with Homebrew:

terminal
brew tap hashicorp/tap
brew install hashicorp/tap/terraform

# verify
terraform -version

Linux, Windows, and manual binary downloads are all covered on the official install page. Once terraform -version prints a version, you're ready.

04 Providers — how Terraform talks to a platform

Terraform's core doesn't know what an S3 bucket or a VPC is. That knowledge lives in providers — plugins that "interact with cloud providers, SaaS providers, and other APIs." Each provider contributes a set of resource types (things you can create) and data sources (things you can read).

So your very first job in any configuration is to declare which provider you need and configure it. Providers are downloaded automatically from the Terraform Registry — the public catalogue of providers and modules.

terraform.tf
# Which providers this configuration needs, and from where.
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"  # address in the Registry
      version = "~> 5.0"          # allow 5.x, not 6.0
    }
  }
}

# How to configure that provider.
provider "aws" {
  region = "eu-south-1"  # Milan
}

Two distinct things here: required_providers declares the dependency and pins its version; the provider "aws" block configures it (region, credentials, etc.). Credentials are normally supplied through the environment (e.g. AWS_PROFILE), not hard-coded.

Source: Providers — HashiCorp Developer

05 The language: blocks & arguments

You just read some Terraform code; now here's the whole grammar. Terraform configurations are written in HCL (HashiCorp Configuration Language), in files ending .tf. There are only two things to recognise:

  • Arguments assign a value to a name: region = "eu-south-1". Format: name = value.
  • Blocks are containers for other content. A block has a type (e.g. resource), zero or more labels (the quoted strings), and a body in { }.
anatomy of a block
resource "aws_s3_bucket" "artifacts" {   ← type + two labels
  bucket = "acme-ml-artifacts"      ← an argument
}                                            ← body in braces

Blocks can nest other blocks, and an argument's value can be a literal or an expression that references other values (we'll use that power in §9). That's genuinely all the syntax there is — everything else is just specific block types.

Source: Configuration Syntax — HashiCorp Developer

06 Resources — the building blocks

A resource is the most important block type: each one describes a single piece of infrastructure you want Terraform to manage. The two labels are the resource type (defined by the provider, e.g. aws_s3_bucket) and a local name you choose (e.g. artifacts) for referring to it elsewhere.

main.tf
resource "aws_s3_bucket" "artifacts" {
  bucket = "acme-ml-artifacts"
}

This says: "there should exist an S3 bucket with this name; I'll call it artifacts within my config." The pair aws_s3_bucket.artifacts is how you'll reference it later. Notice you describe the desired end-state — not "create a bucket". If the bucket already exists and matches, Terraform does nothing.

Source: Resource Blocks — HashiCorp Developer

07 Your first run: init, plan, apply

Put the blocks from §4 and §6 into a directory (the .tf files together form a configuration). Now run the core workflow with three commands.

terraform init — set up the directory

The first command you run on any new or freshly-cloned configuration. It downloads the providers you declared and prepares the working directory. Safe to run repeatedly; it never touches your real infrastructure.

terraform plan — preview

Creates an execution plan: Terraform reads reality, compares it to your config, and prints what it would do — marking each resource + create, ~ change, or - destroy. It changes nothing. This is your safety net — always read it.

terraform apply — make it real

Shows the same plan, then asks you to type yes. On approval it executes the plan and provisions the resources.

terminal
terraform init       # once per directory (and after adding providers)
terraform plan       # preview — safe, read it carefully
terraform apply      # prompts for confirmation, then provisions
And the reverse

terraform destroy deprovisions everything the configuration manages — handy for tearing down temporary or dev environments. Like apply, it shows a plan and asks for confirmation first.

Sources: init · plan · apply · destroy

08 State — Terraform's memory

The moment you ran apply, Terraform wrote a state file (terraform.tfstate). This is the single most important concept to understand early, because almost every surprising Terraform behaviour traces back to it.

State is how Terraform "maps real world resources to your configuration." Without it, Terraform couldn't tell the difference between "this bucket doesn't exist yet" and "this bucket exists and I created it." When you next run plan, Terraform compares three things: your config (desired), the state (what it last knew), and reality (the live API) — and proposes the difference.

Treat state as sensitive

The state file can contain secrets in plaintext (passwords, keys, generated tokens). Never commit terraform.tfstate to git, and don't edit it by hand. §13 shows where it should actually live for a team.

Source: State — HashiCorp Developer

09 References & the dependency graph

Real infrastructure is interconnected, so resources need to reference each other. Recall from §6 that a resource is addressed as <type>.<name>. Append an attribute to read a value off it: aws_s3_bucket.artifacts.id.

When one resource's argument references another resource, Terraform "creates an implicit dependency between the two." It uses these references to build a dependency graph and automatically provisions things in the right order — no manual sequencing.

main.tf
resource "aws_s3_bucket" "artifacts" {
  bucket = "acme-ml-artifacts"
}

resource "aws_s3_bucket_versioning" "artifacts" {
  bucket = aws_s3_bucket.artifacts.id    # ← reference = implicit dependency
  versioning_configuration {
    status = "Enabled"
  }
}

Because the versioning resource references the bucket's id, Terraform knows to create the bucket first. (For the rare case where a dependency exists but isn't expressed through a reference, there's an explicit depends_on argument.)

Source: References to Values — HashiCorp Developer

10 Input variables — parameterise your config

Hard-coding "acme-ml-artifacts" works once. To reuse the same config across environments, lift values into input variables — they "let consumers customise behaviour without altering the source code." Declare with a variable block, read with var.<name>.

variables.tf
variable "bucket_name" {
  type        = string
  description = "Name of the artifacts bucket"
}
main.tf
resource "aws_s3_bucket" "artifacts" {
  bucket = var.bucket_name    # ← value supplied at runtime
}

You supply values in several ways. The most common: a terraform.tfvars file, which Terraform loads automatically.

terraform.tfvars
bucket_name = "acme-ml-artifacts"

Other options: -var 'bucket_name=...' on the command line, any *.tfvars file passed with -var-file, or an environment variable named TF_VAR_bucket_name.

Source: Input Variables — HashiCorp Developer

11 Output values — expose results

Variables feed values in; outputs surface values out. After an apply, Terraform often generates data you need — an ARN, an endpoint, an IP. An output block exposes it.

outputs.tf
output "artifacts_bucket_arn" {
  value = aws_s3_bucket.artifacts.arn
}

Outputs are printed in the CLI after apply, and they're how a child module exposes data to its parent (you'll meet modules in §14) and how other configurations read your values via remote state (§13). Retrieve them anytime with terraform output.

Source: Output Values — HashiCorp Developer

12 Data sources — reading existing infrastructure

Resources create things. Sometimes you instead need to read something that already exists — a VPC created elsewhere, the latest AMI, your account ID. That's a data source: it "fetches data from the provider but does not create or modify resources."

Declare with a data block; reference with data.<type>.<name>.<attribute>.

main.tf
# Look up an existing VPC by tag (read-only).
data "aws_vpc" "main" {
  tags = { Name = "acme-prod" }
}

# Use the looked-up value when creating a new resource.
resource "aws_security_group" "redis" {
  vpc_id = data.aws_vpc.main.id    # ← reference into the data source
}

Just like resource references, a data-source reference creates a dependency: Terraform reads the VPC before creating the security group. Data sources keep configs flexible — you don't have to hard-code IDs that live outside your control.

Source: Data Sources — HashiCorp Developer

13 Remote state & backends — working as a team

By default the state file (§8) sits on your laptop. That breaks the moment a teammate needs to apply, or two people apply at once. The fix is a backend: it "defines where Terraform stores its state data files."

A remote backend (S3, Azure Blob, GCS, HCP Terraform, …) gives you three things a local file can't:

  • Shared state — everyone reads/writes the same source of truth.
  • Locking — while one apply runs, others are blocked, preventing two people from corrupting state simultaneously.
  • Safety — sensitive state lives in managed, access-controlled, versioned storage instead of on laptops or in git.

You configure it inside the terraform block you first saw in §4:

backend.tf
terraform {
  backend "s3" {
    bucket = "acme-terraform-state"
    key    = "shared-resources/terraform.tfstate"  # path within the bucket
    region = "eu-south-1"
  }
}

The key lets different parts of your infrastructure (e.g. back-end, shared-resources, rms) keep separate state files in one bucket. Changing the backend requires re-running terraform init.

Source: Backend Configuration — HashiCorp Developer

14 Modules — group & reuse

Everything so far lived in one directory. That directory is itself a module — the root module. A module is simply "a collection of resources that Terraform manages together," i.e. a folder of .tf files. The power comes from calling one module from another to avoid copy-pasting.

A reusable module exposes input variables (§10) as its knobs and output values (§11) as its results — that's why those came first. You call it with a module block, pointing source at its location and passing the variables as arguments:

main.tf (root module)
module "redis" {
  source = "./modules/elasticache-redis"   # local folder, Registry, or git URL

  # these map to variable blocks inside the module
  node_type = "cache.t4g.micro"
  vpc_id    = data.aws_vpc.main.id
}

You'd then read its results via module.redis.<output_name>. Modules are how teams standardise infrastructure: write a vetted "Redis cluster" or "VPC" once, reuse it everywhere. The Terraform Registry hosts thousands of ready-made community and verified modules.

The shape of a real repo

This is why production Terraform repos split into per-environment or per-component directories, each a root module with its own state key, calling shared child modules under a modules/ folder. You now have every concept needed to read one.

Source: Modules — HashiCorp Developer

15 Good habits from day one

HashiCorp publishes an official Style Guide. The two habits worth adopting immediately:

  • Run terraform fmt before every commit. It rewrites your files into the canonical format, so diffs stay clean and style never gets debated in review. A git pre-commit hook automates it.
  • Use the conventional file names. A standard module splits its blocks across files so anyone can find things instantly:
FileHolds
main.tfResources and data sources
variables.tfAll variable blocks (alphabetical)
outputs.tfAll output blocks (alphabetical)
providers.tfProvider configuration
terraform.tfrequired_providers / version constraints
backend.tfBackend / remote-state config

Terraform doesn't care how you split files — it reads every .tf in the directory as one config — but these conventions make a codebase predictable. Also keep terraform validate in your loop: it checks a config is internally consistent, offline, before you plan.

Source: Style Guide — HashiCorp Developer
