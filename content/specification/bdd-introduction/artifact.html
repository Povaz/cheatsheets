1What BDD is — and isn't

Behaviour-Driven Development (BDD) is a way of building software where the team agrees on concrete examples of how the system should behave before writing code, then uses those same examples to drive and verify the implementation.

The most common misconception is that BDD is a testing technique. The Cucumber team is blunt about it: BDD is "collaborative requirements analysis combined with test-driven development." Tests are a by-product, not the point.

BDD isBDD is not
A collaboration practice: product, dev, and QA agreeing on examples up frontA test-automation tool bolted on after coding
Examples written in business language everyone can readScripts only engineers can decipher
Done before implementation, guiding itRetroactive documentation of what was built
Watch out Teams that skip the conversation and hand "the BDD tests" to QA after the code ships have abandoned BDD entirely — they keep the syntax and lose the value. That failure mode, not the method, is behind most BDD criticism.

Behaviour-Driven Development · BDD Is Not Test Automation

2The three practices

Day-to-day BDD is a small, iterative loop over one upcoming change at a time:

DiscoveryTalk through concrete examples to "explore, discover and agree" on what's expected.
→
Formulation"Document those examples in a way that can be automated, and check for agreement."
→
AutomationImplement the behaviour, "starting with an automated test to guide the development."
  • Discovery produces shared understanding — the conversation is the deliverable.
  • Formulation produces a business-readable, machine-executable specification (section 3 covers the language).
  • Automation produces code guided by those examples — each example becomes a test that fails until the behaviour exists.
Tip Work one small slice at a time. The loop is designed to run in hours, not sprints — one example discovered, formulated, and automated before the next.

Behaviour-Driven Development

3Gherkin, the language of Formulation

Gherkin is the plain-text language those examples are formulated in. It reads as structured English, but a runner can parse and execute it. A file of Gherkin is a feature file (.feature).

KeywordRole
Feature:Names the capability; a short free-text narrative can follow.
Scenario:One concrete example of behaviour, with a single pass/fail outcome.
GivenThe world before the action — context, never behaviour under test.
WhenThe single triggering action or event.
ThenThe observable outcome to assert.
And / ButContinue the previous keyword without repeating it.
Background:Given-steps shared by every scenario in the file.
Scenario Outline:A scenario template, run once per row of its Examples table; <placeholders> are replaced with the row's values before step matching.
@tagA label on a Feature or Scenario, used to select what to run.

A complete feature file — specifying lesson attachments in an online course platform:

e2e/features/lesson-attachments/attach-file-to-lesson.feature

@LessonAttachments
Feature: Attach a file to a Lesson

  Background:
    Given the user is authenticated as a Course Author
    And a lesson exists in a module locked for review

  Scenario Outline: AC-31.1 Uploading a file, with or without a comment, attaches it
    Given the Course Author is viewing the Lesson
    When the Course Author uploads the file "<file>" with comment "<comment>"
    Then "<file>" appears in that lesson's Attachment list
    And the listed Attachment shows the comment "<comment>"
    And the listed Attachment shows the Course Author as uploader and records the upload time

    Examples:
      | file                | comment             |
      | syllabus-2026.pdf   | week 3 reading list |
      | grading-rubric.xlsx |                     |

Note the shape: exactly one When; the outline replaces an "optionally with a comment" branch by making the comment data — a value in one row, empty in the other. Same behaviour, varied data.

4Declarative scenarios: what, not how

The core style rule of Formulation: "Your scenarios should describe the intended behaviour of the system, not the implementation." Scenarios written as UI walkthroughs are called imperative; scenarios written as behaviour are declarative.

Imperative — brittle, unreadable as documentation

When I click the file input
And I select "syllabus-2026.pdf" from the picker
And I type "week 3 reading list" into the comment box
And I click the "Upload" button

Declarative — stable, reads as behaviour

When the Course Author uploads the file "syllabus-2026.pdf" with comment "week 3 reading list"

Imperative steps are "so closely tied to the mechanics of the current UI" that every UI change forces a spec change. Declarative steps survive redesigns — the mechanics move into code (section 5) where they belong.

Tip Declarative ≠ vague. Hide the mechanics (clicks, fields, endpoints); keep the data concrete ("syllabus-2026.pdf", "week 3 reading list"). A scenario with no real values can't fail meaningfully.

Writing Better Gherkin

5Step definitions: the glue

A step definition is the code a Gherkin step matches against — the bridge between the specification and the system. The runner matches each step's text to a pattern and executes the attached function. All the "how" that section 4 evicted from the scenarios lives here.

TypeScript, playwright-bdd — one step, two layers of "how"

// step-definitions/lesson-attachments/given.ts
Given('a lesson exists in a module locked for review', async ({ api }) => {
  // orchestrates two API calls behind one declarative step:
  const id = await api.createLesson();   // authenticated POST /api/lessons/
  await api.lockModule(id);              // authenticated POST /api/lessons/<id>/lock-module/
});

Two conventions keep this layer healthy:

  • Organize by domain concept — "a good rule of thumb is to have one file for each major domain object", and "refactor step definitions into helper methods for greater modularity and reuse." A step stays one line; helpers do the work.
  • Set up state through the API, never the UI. A Given is scaffolding, not behaviour under test. Driving setup through screens creates false dependencies: an unrelated page change breaks every scenario that merely passed through it.
Watch out If a precondition has no API to build it with, that's a finding, not a dead end — flag the missing endpoint (or add a test-only one) rather than working around it through the UI.

6Anti-patterns

The recurring failure modes, from the Cucumber docs and the Cucumber core team's own discussion (Aslak Hellesøy, Matt Wynne, Steve Tooke):

Anti-patternWhy it hurts / the fix
Features written after the codeBDD run backwards — no Discovery, no shared understanding, just expensive tests.
Conjunction steps"Don't use steps that combine a bunch of different things" — too specialised, impossible to reuse. Split; And exists for a reason.
Feature-coupled step definitionsSteps reusable only inside one feature duplicate code across the suite. Organize by domain concept instead (section 5).
Multiple When stepsTwo triggers means two behaviours — an indicator "you should split this scenario."
Testing several things in one scenarioBlurs the essence; on failure you can't tell which behaviour broke.
Conditional / optional steps"Optionally…", "…if one was given" branches inside one scenario — no single pass/fail. Make the variation an Examples row (section 3), or split when the behaviour differs.
Scenario Outline overuse"Easily… overused and lead to slow tests" — especially through the UI. Keep UI outlines to a few rows; push data sweeps to cheaper layers (section 8).
Incidental detailValues that don't affect the outcome ("a user named Bob with a red hat") bury the signal. Every detail present should matter.

7Living documentation

Once the examples execute, the specification and the test suite are the same artifact. That's living documentation: "the code reflects the documentation, and the documentation reflects the team's shared understanding." Ordinary docs drift silently; living documentation that drifts fails the build.

This makes the feature files the single source of truth for behaviour. Two consequences:

  • There is exactly one canonical place a behaviour is specified. Derived copies (exports, generated files) are fine — as long as they are mechanically derived and checked for drift, never hand-edited.
  • A new team member reads the feature files, not the code, to learn what the system does.
Tip Guard the derivation in CI: regenerate the executable files from the canonical spec and fail if git diff is non-empty. Drift becomes a build failure instead of a discovery six months later.

Behaviour-Driven Development

8Where the scenarios run: the test pyramid

Executable scenarios still need a layer to execute against. The test pyramid is the sizing rule: "write lots of small and fast unit tests… some more coarse-grained tests and very few high-level tests that test your application from end to end."

UI / e2e — very few
Service / API — some
Unit — lots

Higher = slower, more brittle, more expensive to maintain.

Acceptance scenarios are not chained to the top: "there's technically no need to write acceptance tests at the highest level" — the guidance is to "push your tests as far down the test pyramid as you can." A practical split:

  • Happy path → one scenario through the UI, proving the whole slice hangs together.
  • Sad paths (validation, limits, permissions) → same Gherkin discipline, executed at the API layer — no browser, same confidence, a fraction of the cost.

9BDD in a real project

How the concepts above map onto a worked example — the online course platform whose Lesson Attachments feature ran through sections 3–5:

Canonical specHLDD us-*.md — fenced Gherkin with backticked Dictionary terms
→
Generatorstrips fences, backticks, trailing commas; stamps the Context @tag
→
.featuree2e/features/ — the runner-executed derivative
→
playwright-bddstep definitions run on @playwright/test
  • Formulation happens in the HLDD (per-Context folders, AC-X.Y codes). The .feature files are generated, never hand-edited — section 7's drift-check applies.
  • Given-steps compose the real API: authenticated create-lesson + lock-module requests behind one declarative step (section 5's example).
  • Locators follow Playwright's guidance to prioritize "user-facing attributes and explicit contracts" — getByRole / getByLabel first; data-testid is the escape hatch, not the default. Kent C. Dodds: if you need it everywhere, "it's because your app is inaccessible."
  • Isolation — each scenario gets a fresh browser context and its own API-created lesson; Playwright's rule that "each test should be completely isolated… and run independently."
  • Environment is owned by the harness, never by a Given: the app stack (API, database, object storage) runs locally via docker compose; Playwright's webServer starts the frontend (reuseExistingServer locally, fresh in CI).
