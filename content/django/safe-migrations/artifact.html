Deployment-Safe Django Migrations

A rolling deploy runs old and new code at the same time for a short window. A migration is "safe" only if it survives that window. This guide explains the two libraries that manage it — django-safemigrate (you mark each migration) and django-syzygy (it splits and marks them for you) — and ends with one worked workflow on Postgres.

01

The coexistence window

A zero-downtime deploy doesn't swap all servers at once. It rolls out gradually, so for a period the old code and the new code both run against the same database. Migrations usually run once, at the edge of that window — not on every server.

That creates a constraint Django's autodetector does not reason about: the schema after a migration has to work for both versions of the code that touch it during the rollout.

  • Run the schema change before the new code → the still-running old code meets a schema it doesn't expect.
  • Run it after the new code → the new code spent the rollout talking to a schema that wasn't there yet.

So a single change can be unsafe in both directions. The fix is to decide, per migration, which side of the code swap it belongs on — and to split the changes that belong on both.

02

Three migration stages

Both libraries use the same three-way vocabulary for the slots in Fig.1. Learn it once; the rest of the guide reuses it.

StageMeaningTypical operations
pre-deploySafe to apply before the new code ships. Additive: old code keeps working because nothing it relies on is removed.AddField(null=True), CreateModel, add index
alwaysSafe before and after — it changes nothing the database enforces.help_text, verbose_name, choices
post-deploySafe only after all old code is gone. Destructive: removes something the old code might still touch.RemoveField, DeleteModel

The stage is mostly a property of the operation type, so you can decide it up front rather than discovering it after makemigrations. A migration that fits none of these almost always means one of two things:

  1. You bundled several model edits into one run — split your model edits so each makemigrations produces a single-stage migration; or
  2. You hit an operation that is internally two-phase. There are essentially two of those — adding a column with a default, and removing a column — and both have a known split (covered in §3–§5).
Tip · the naming differs, the idea doesn't

safemigrate calls these Safe.before_deploy / Safe.always / Safe.after_deploy. syzygy calls them Stage.PRE_DEPLOY / (prerequisite) / Stage.POST_DEPLOY. Same three slots.

03

The db_default primitive

You need this one Django feature before either tool makes sense, because it's how both of them make the hardest case safe.

When you add a column with a Python default, Django emits SQL that adds the column with a database default and then immediately drops that default:

SQL Django generates for AddField(default=…) 
ALTER TABLE "order" ADD COLUMN "code" varchar NOT NULL DEFAULT 'TBD';
ALTER TABLE "order" ALTER COLUMN "code" DROP DEFAULT;   -- ← the problem

Once the default is dropped, the old code still running during the rollout inserts rows without that column — and the NOT NULL column with no default rejects the insert. That is exactly a change that is unsafe before deploy.

db_default (Django 5.0+) fixes the root cause by persisting the default in the database schema, so the DROP DEFAULT never happens and old-code inserts keep working. Per the Django docs: if both db_default and default are set, default wins in Python, while db_default stays at the database level and is used for inserts made outside the ORM or when adding a field in a migration.

models.py — survives a rolling deploy 
class Order(models.Model):
    code = models.CharField(max_length=12, db_default="TBD")
Gotcha · DatabaseDefault on unsaved instances

If a field has db_default but no Python default, reading it on an unsaved instance returns a DatabaseDefault wrapper object, not the value — so if order.code: can behave unexpectedly before .save(). Setting both default and db_default avoids this.

04

django-safemigrate

Philosophy: you label each migration; the tool enforces the labels at deploy time. It adds one new command, safemigrate, and otherwise leaves Django alone.

Install

pip install django-safemigrate

# settings.py
INSTALLED_APPS = [ ..., "django_safemigrate" ]

Mark each migration

Set a safe attribute on the Migration class. As of v6.0 the values are callables:

a migration that adds a column 
from django_safemigrate import Safe

class Migration(migrations.Migration):
    safe = Safe.before_deploy()
    operations = [ ... ]
  • Safe.before_deploy() — only safe before the code change (e.g. add a field).
  • Safe.after_deploy() — only safe after the code change (e.g. remove a field). Accepts delay=timedelta(days=7) to postpone how soon safemigrate will apply it, measured from when the migration is first detected.
  • Safe.always() — safe either side; this is the default when no safe is given.

Run it

  1. Before rolling out code, run python manage.py safemigrate. It applies the before_deploy (and always) migrations. If a before_deploy migration depends on an after_deploy one that hasn't run, it refuses and errors — that's the guardrail.
  2. After the code is fully deployed, run the normal python manage.py migrate, which applies everything that's left, including the after_deploy migrations.
Warning · after_deploy doesn't run on first sight

A Safe.after_deploy migration will not run the first time safemigrate encounters it — by design, it waits for the post-deploy migrate. Make sure your pipeline actually runs that second migrate, or these accumulate and later block a before_deploy migration that depends on them.

Keep labels honest

  • Pre-commit hook — checks every migration carries a safe value: 
    # .pre-commit-config.yaml
repos:
  - repo: https://github.com/ryanhiebert/django-safemigrate
    rev: "6.0"
    hooks:
      - id: check
  • SAFEMIGRATE = "nonstrict" — for dev: run whatever isn't blocked instead of erroring.
  • SAFEMIGRATE = "disabled" — make safemigrate behave like plain migrate.
Gotcha · attribute vs. callable

Older examples (including the README quickstart) show safe = Safe.before_deploy with no parentheses. v6.0 turned the values into callables and added the delay option, so prefer the called form Safe.before_deploy(). Pin django-safemigrate>=6.0.

05

django-syzygy

Philosophy: the tool infers the stage and rewrites unsafe operations for you at generation time. It overrides Django's makemigrations and migrate commands rather than adding a separate one.

Install

pip install django-syzygy

# settings.py
INSTALLED_APPS = [ ..., "syzygy" ]
Warning · it overrides the built-in commands

Adding "syzygy" replaces the default makemigrations and migrate (the framework gives no cleaner injection point). If you combine it with another migration package that also overrides those commands, you must write your own command classes inheriting from both. This is why syzygy is an alternative to safemigrate, not an add-on.

How it decides the stage

A migration is treated as a prerequisite (pre-deploy) unless it contains a destructive operation or sets stage = Stage.POST_DEPLOY. An explicit stage attribute always overrides the heuristic.

What it rewrites automatically

  • Field addition. It splits the unsafe add-then-drop-default into a pre-deploy column add and a post-deploy step that drops the database default. On Django 5.0+ it instead emits a vanilla AddField + AlterField that lean on db_default (§3).
  • Field removal. It generates a pre-deploy step that makes the column safe for old code (adds a DB default or makes it nullable) and a post-deploy RemoveField. On 5.0+ the pre-deploy step is a vanilla AlterField using db_default.

Run it

python manage.py migrate --pre-deploy   # before rolling out code
# … deploy the code …
python manage.py migrate              # after; applies postponed migrations

Guardrails

  • System checks error on any migration that mixes operations of different inferred stages, forcing you to annotate stage explicitly. syzygy refuses to guess on ambiguous changes by design.
  • CI check — run migrate --pre-deploy against a database that has only the target-branch changes; if the branch can't be cleanly partitioned it raises AmbiguousPlan with the resolution paths: 
    git checkout main && python manage.py migrate
git checkout add-feature && python manage.py migrate --pre-deploy
  • Third-party migrations you can't edit: configure MIGRATION_STAGES_FALLBACK, MIGRATION_STAGES_OVERRIDE, and MIGRATION_THIRD_PARTY_STAGES_FALLBACK (defaults to pre-deploy).
  • Multi-cluster on one database — migrate --quorum N coordinates clusters so each migration applies exactly once, after configuring MIGRATION_QUORUM_BACKEND (e.g. the cache-based CacheQuorum backed by Redis/Memcached). Relevant if your AWS setup runs the same DB across clusters.
Tip · it won't make every change atomic

syzygy is explicit that some changes are inherently un-atomic and it won't pretend otherwise — and that it does not minimize database lock contention. For that, see django-pg-zero-downtime-migrations (Sources).

Side by side

Both solve the §1 problem; they differ in who decides the stage and how much they touch Django.

 django-safemigratedjango-syzygy
Who sets the stageYou, per migration (safe = …)Inferred from operations; override with stage = …
Splits unsafe opsNo — you stage your model editsYes — rewrites AddField / RemoveField
CommandsAdds safemigrate; leaves migrate intactOverrides makemigrations & migrate

Tip · which to reach for

Want explicit control and minimal interference with Django's commands → safemigrate. Want the generation step to produce correctly-staged, pre-split migrations with little manual marking → syzygy. Pick one; don't run both.

07

Worked workflow

Same change, on Postgres, in a DRF service. Starting model:

models.py — before 
class Order(models.Model):
    customer = models.ForeignKey(Customer, on_delete=models.CASCADE)
    legacy_ref = models.CharField(max_length=32)   # to be removed

Goal: add a required confirmation_code and remove legacy_ref. A column drop is unsafe before deploy — old code still issues SELECT/INSERT/UPDATE that name the column — and safe only once that code is gone, so RemoveField is a post-deploy operation (§2). Adding a required column is unsafe before deploy unless db_default carries the default into the schema (§3). The two operations land on opposite sides of the code swap, so the change spans both phases of the rollout — pre-deploy and post-deploy — not because either migration is unsafe in both directions. Here are both tools doing it.

django-safemigrate

  1. Edit model: add confirmation_code = CharField(max_length=12, db_default="TBD"); set legacy_ref to null=True and stop reading/writing it in code & serializers.
  2. makemigrations → mark both the AddField and the legacy_ref→nullable AlterField Safe.before_deploy().
  3. Deploy A: safemigrate (adds the column, makes legacy_ref nullable) → roll out the code → migrate.
  4. Edit model: delete legacy_ref.
  5. makemigrations → mark the RemoveField Safe.after_deploy().
  6. Deploy B: safemigrate (no-op pre) → roll out code with the field gone → migrate drops the column.

django-syzygy

  1. Edit model: add confirmation_code = CharField(max_length=12, db_default="TBD"); remove legacy_ref from the model and code in the same change.
  2. makemigrations → syzygy emits correctly-split, stage-tagged operations (pre-deploy add & the pre-deploy half of the removal; post-deploy RemoveField). No manual marking.
  3. Deploy: migrate --pre-deploy → roll out the code → migrate applies the postponed drop.
  4. If the two changes can't be cleanly ordered, migrate --pre-deploy raises AmbiguousPlan and you annotate stage — caught in CI before merge.
Gotcha · dropping a NOT NULL column needs a nullable step first

Once legacy_ref leaves the model, Django stops listing it in INSERTs — so a NOT NULL column with no default rejects writes from the new code during the rollout. The column must be made nullable (or given a db_default) before the drop. That extra pre-deploy step is why the safemigrate path here takes two releases; syzygy generates the same step automatically (the "pre-deploy half") and fits both phases into a single deploy. If legacy_ref were already null=True, the removal would be a clean one-step post-deploy RemoveField in both tools.

Warning · the column lingers between deploys

With either tool, legacy_ref still exists in the database during Deploy A / the pre-deploy phase. Old code must already have stopped depending on it before the post-deploy drop runs — that's the whole point of separating the phases. Don't collapse them.
