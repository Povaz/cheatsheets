Chapter 1Routing & ViewSets

From URL to handler: one class per resource, URLs generated for you, and the helpers the default handlers are built on.

1ViewSet & ModelViewSet

Plain Django maps one URL to one view. DRF's ViewSet inverts that: one class per resource, with methods named after actions rather than HTTP verbs. The six standard actions cover CRUD:

ActionVerb + pathMeaning
listGET /api/orders/all rows (usually paginated)
retrieveGET /api/orders/{id}/one row
createPOST /api/orders/insert
updatePUT /api/orders/{id}/full replace
partial_updatePATCH /api/orders/{id}/edit some fields
destroyDELETE /api/orders/{id}/delete → 204

ModelViewSet pre-wires all six. You declare two things — which rows, and which serializer (the component that translates rows to/from JSON; Chapter 2 owns it):

class OrderViewSet(viewsets.ModelViewSet):
    queryset = Order.objects.all()
    serializer_class = OrderSerializer
  • Base ViewSet: no behavior — you write each action method yourself.
  • ModelViewSet = generic viewset + all CRUD mixins: full resource from two attributes.
  • Override any single action method to customize just that endpoint; the rest stay stock.
Tip

Keep viewsets thin: parse → call a service function → return Response. Rules that change multiple rows together belong in a service module, not in the view.

2Router wiring

A viewset declares actions but no URLs. A router inspects the class and generates the URL patterns in urls.py:

# urls.py
router = DefaultRouter()
router.register(r'orders', OrderViewSet, basename='order')
router.register(r'files', DocumentViewSet, basename='document')

urlpatterns = [
    path('api/', include(router.urls)),
]
  • Each register(prefix, viewset) yields two patterns: the list route /api/orders/ (list + create) and the detail route /api/orders/{pk}/ (retrieve / update / partial_update / destroy).
  • basename seeds the URL names for reverse() (order-list, order-detail); required when the viewset has no queryset attribute to infer it from.
  • router.urls is a plain list of Django URL patterns. At request time it is Django's URL dispatcher doing the matching — DRF adds no routing runtime of its own.
Tip

DefaultRouter adds a browsable API root view at the include prefix; SimpleRouter generates only the resource routes.

DRF — Routers

3get_object() & get_object_or_404

Detail routes carry a pk captured by the URL pattern. Two helpers turn it into a row — or into a clean 404:

# inside a viewset handler
order = self.get_object()            # 404 if the pk matches nothing

# anywhere else
order = get_object_or_404(Order, pk=pk)
  • self.get_object() reads the pk from the URL kwargs, filters the viewset's queryset, runs object-level permission checks, and raises Http404 on no match. Every default detail handler (retrieve, update, destroy) calls it first.
  • get_object_or_404(Model, pk=pk) is the Django shortcut with the same fetch-or-404 contract, minus the viewset context — for code paths outside the default queryset flow (custom actions, plain views).
  • Http404 raised anywhere in the request cycle is converted by Django into an HTTP 404 response — handlers never build the error body by hand.

4@action — extra endpoints

Routers only know the six CRUD actions. @action registers a non-CRUD endpoint on the viewset; the router picks it up automatically. Here: list every document attached to one order.

class OrderViewSet(viewsets.ModelViewSet):
    queryset = Order.objects.all()
    serializer_class = OrderSerializer

    @action(detail=True, methods=['get'], url_path='files')
    def files(self, request, pk=None):
        get_object_or_404(Order, pk=pk)        # 404 before any work
        docs = document_service.list_for_order(pk)
        return Response(DocumentSerializer(docs, many=True).data)
  • detail=True hangs the route off the detail URL — scoped to one object: GET /api/orders/{pk}/files/.
  • detail=False hangs it off the collection: /api/orders/export/ — for anything about the set, not one row (bulk import, search, stats).
  • methods defaults to ['get']; url_path defaults to the method name; url_name seeds reverse().

5destroy vs perform_destroy

Every write action splits into an HTTP layer and a work hook. destroy(request, …) is the HTTP handler: it resolves the object from the URL and returns 204 No Content; in between it calls perform_destroy(instance), which is one line — instance.delete().

Override the hook to swap what deletion means, keeping DRF's URL parsing and response contract:

def perform_destroy(self, instance):
    instance.archived = True
    instance.save(update_fields=['archived'])   # soft delete

Override the handler when the whole act is delegated — e.g. a service that must detach related rows in the same transaction:

def destroy(self, request, *args, **kwargs):
    instance = self.get_object()              # keep the 404 contract
    order_service.delete_order(instance.id)   # service owns atomicity
    return Response(status=status.HTTP_204_NO_CONTENT)
Gotcha

An overridden destroy that skips self.get_object() never 404s: an unknown id flows into the service and silently no-ops. Call it first, always.

The same split holds across the board: create/perform_create, update/perform_update.

6Testing: APIClient + force_authenticate

DRF ships a test client that speaks the same wire as production but lets tests skip authentication mechanics and target behavior:

class OrderFilesAPITests(TestCase):
    @classmethod
    def setUpTestData(cls):
        cls.user = User.objects.create_user('reviewer')
        cls.order = Order.objects.create(number='SO-1042')

    def setUp(self):
        self.client = APIClient()
        self.client.force_authenticate(self.user)

    def test_lists_attached_documents(self):
        res = self.client.get(f'/api/orders/{self.order.id}/files/')
        self.assertEqual(res.status_code, 200)
  • force_authenticate(user) bypasses the real auth pipeline — no token issuance, no Authorization header. Every following request counts as that user.
  • APIClient responses expose .status_code and parsed .data — assert on both.
  • Django's TestCase wraps each test in a transaction rolled back on exit: DB writes never leak between tests.
  • setUpTestData runs once per class — put shared rows there for speed. setUp runs per test — put per-test state (like the client) there.

Chapter 2Serializers

From handler to wire: the layer that renders, parses, and validates JSON — and the one upload shape that is not JSON.

7Serializers & ModelSerializer

Chapter 1 moved requests to handlers. Serializers are DRF's JSON I/O layer, working in both directions:

  • Out (read): instance → dict of primitives (.data) → rendered JSON.
  • In (write): parsed payload → validation → .validated_data → saved row.

ModelSerializer introspects the model and generates one serializer field per model field — types, lengths, and required flags come from the model definition:

class OrderSerializer(serializers.ModelSerializer):
    class Meta:
        model = Order
        fields = ['id', 'number', 'status', 'placed_at']
# out: instance -> primitives
OrderSerializer(order).data          # {'id': …, 'number': 'SO-1042', …}

# in: payload -> validated data -> row
ser = OrderSerializer(data=request.data)
ser.is_valid(raise_exception=True)   # auto-400 with per-field errors
order = ser.save()
  • is_valid(raise_exception=True) makes DRF return the 400 with a per-field error dict — no manual error body.
  • In a ModelViewSet you rarely drive it by hand: the stock create/update actions do — serializer_class from §1 is where it plugs in.
Warning

Prefer an explicit fields allowlist over '__all__': new model columns should reach the wire deliberately, not by default.

DRF — Serializers

8SerializerMethodField

ModelSerializer mirrors the model. SerializerMethodField lets the wire shape diverge from it: the value is computed by a get_<name>(self, obj) method on the serializer.

Example: Document attachments point at either an Order or an Invoice (an entity_type discriminator). Invoice rows should carry an invoice_number label the model itself doesn't have:

class AttachmentSerializer(serializers.ModelSerializer):
    invoice_number = serializers.SerializerMethodField()

    class Meta:
        model = DocumentAttachment
        fields = ['id', 'entity_type', 'entity_id',
                  'attached_at', 'invoice_number']

    def get_invoice_number(self, obj):
        if obj.entity_type != 'invoice':
            return None
        return obj.target.number
  • Read-only by definition — never parsed on input.
  • Naming contract: field x ↔ method get_x.
  • Runs per row on output; any Python is fair game — lookups, formatting, joins across relations.
Gotcha

Serializer fields are class-level declarations, evaluated once at import — you cannot declare a field for some rows and not others. The method runs for every row and returns None where it doesn't apply; dropping the key entirely is the next section's job. And when the method touches related rows on a list endpoint, watch for N+1 queries.

DRF — SerializerMethodField

9to_representation

to_representation(instance) is the per-row hook DRF calls to turn an instance into the output dict — the last stop before JSON encoding, and the place for conditional shaping. Continuing §8: strip the invoice-only key from non-invoice rows so the wire stays minimal.

def to_representation(self, instance):
    data = super().to_representation(instance)
    if data['entity_type'] != 'invoice':
        data.pop('invoice_number', None)    # keep the wire minimal
    return data
  • Pattern: super() first — it builds the dict from all declared fields (method fields included) — then mutate and return.
  • The add-then-strip idiom: declare per-variant label fields for every row (§8), then pop the keys that don't apply per row here. Each entity_type gets a typed shape; consumers never see null noise.
  • The mirror hook for input is to_internal_value — same idea, parse direction.

10Multipart bracket-indexed upload

File bytes cannot ride in a JSON body. Uploads travel as multipart/form-data; DRF's MultiPartParser merges the form fields and the files into request.data. The wire shape for "upload one document, attach it to two entities":

POST /api/files/
Content-Type: multipart/form-data

file:                      <binary bytes>
title:                     "Signed delivery note"
attach_to[0]entity_type:   "order"
attach_to[0]entity_id:     "9f3c…"
attach_to[1]entity_type:   "invoice"
attach_to[1]entity_id:     "7a01…"
  • Bracket-indexed keys (attach_to[i]key) are how a nested list crosses form encoding — multipart has no native nesting. DRF's form parsing (used by ListField / nested serializers with many=True) reassembles them into a list of dicts for validation.
  • This is exactly what the browser produces: new FormData() + fd.append('attach_to[0]entity_type', 'order') — no client-side serialization tricks.
  • Enable with parser_classes = [MultiPartParser, FormParser] on the view (or the global default). The file arrives as an UploadedFile in request.data['file'], ready for a serializer FileField.
Warning

It is not a JSON body. Mixing raw file bytes into JSON is only possible via base64 bloat; the native path is FormData → multipart → MultiPartParser.

Tip — testing uploads

Build the file with SimpleUploadedFile and post with §6's client using format='multipart':

pdf = SimpleUploadedFile('note.pdf', b'%PDF-1.4 …',
                         content_type='application/pdf')
res = self.client.post('/api/files/', {
    'file': pdf,
    'title': 'Signed delivery note',
    'attach_to[0]entity_type': 'order',
    'attach_to[0]entity_id': str(self.order.id),
}, format='multipart')
