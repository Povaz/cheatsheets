Chapter 01

Service layer & transactions

Django gives you models, views, and URLs — and deliberately says nothing about where multi-model write logic goes. This chapter fixes the two conventions everything later builds on: where writes live, and how they stay all-or-nothing.

1.1The service layer — a convention, not a framework feature

Neither Django nor DRF prescribe a service layer. It is a project-level decision: views parse requests and return responses, services orchestrate multi-model writes, models stay focused on "what is a row."

LayerOwnsNever does
ViewHTTP parsing, status codes, serializationBusiness rules, multi-row writes
ModelFields, constraints, per-row helpersCross-model orchestration
Service (services/*.py)How multiple rows change together: validation, atomicity, cleanupHTTP concerns

1.2The single seam

When multiple callers can trigger the same write — an API view, an admin action, a management command — route them all through one service function. That function becomes the single seam: validation, atomicity, and cleanup live in one place.

In the running example, the only sanctioned way to create a Document plus its DocumentAttachment rows is document_service.create_with_attachments. The eligibility rule "only MANUAL refunds accept attachments" is checked there, before anything is written.

# services/document_service.py — every caller lands here
@transaction.atomic
def create_with_attachments(uploaded_file, title, attach_to, user):
    targets = [_resolve_or_raise(t) for t in attach_to]  # validate BEFORE any write
    doc = Document.objects.create(title=title, uploaded_by=user)
    doc.file.save(title, uploaded_file)   # bytes go to the storage backend, not the DB
    try:
        _create_attachment_rows(doc, targets)
    except Exception:
        doc.file.delete(save=False)       # atomic() cannot roll back a blob
        raise                             # DB rows roll back here
    return doc
  • Rejections are free rollbacks. Validation runs before the first write; when it fails, no blob exists and no row exists — nothing to clean up.
  • Future entry points inherit every rule. A bulk-upload endpoint added next year routes through the same function and gets the MANUAL-refund check without touching it.
  • Rules never scatter. The tempting alternative — repeating the check in each view — drifts the first time one copy is edited.

1.3transaction.atomic

The primitive under the seam. It is both a context manager and a decorator that wraps a block in a database transaction:

  • Commit on clean exit, rollback on exception.
  • Nestable — inner blocks become savepoints: an inner rollback does not abort the outer transaction.
  • Two granularities. Per request: ATOMIC_REQUESTS=True wraps each view in one transaction — right when the user expects all-or-nothing. Per iteration: a batch job opens one atomic() per item, so one bad item cannot roll back its successful peers (Chapter 5 shows this live).
Gotcha

atomic() rolls back rows, not side effects. A file written to disk or object storage is never enrolled in the DB transaction. The seam must track what it wrote and delete the orphaned blob itself — that is the doc.file.delete(save=False) line above. Chapter 4 makes the storage side concrete.

Chapter 02

Querying without surprises

Two skills: pushing computation into SQL, and keeping the query count constant no matter how many rows come back.

2.1Computed columns: .annotate() + database functions

.annotate() attaches a SQL-computed column to each queryset row. Because the value is computed in SQL, it is usable inside .filter() on the same query — no round-trip to Python.

The expressions come from django.db.models.functions — thin wrappers around SQL functions. Substr(field, pos, length) translates to SUBSTRING(field FROM pos FOR length).

# invoice PDF "AB12-0042.pdf" → the Order whose code starts with "AB12"
matches = (Order.objects
    .annotate(code_prefix=Substr('order_code', 1, 4))  # computed in SQL…
    .filter(code_prefix=prefix))                          # …so filterable in SQL
Gotcha — 1-indexed

PostgreSQL SUBSTRING is 1-indexed, and Substr passes pos straight through. Substr('order_code', 1, 4) means "the first 4 characters" — not Python's [1:5].

Tip

order_code__startswith=prefix would use the same B-tree index. Prefer Substr when the intent is "these exact character positions" — it also generalizes to substrings not anchored at position 1, where __startswith cannot go.

2.2The N+1 query problem

Listing N rows, then reading one related field per row, triggers one extra query per row: 1 query for the list + N follow-ups, when 1 or 2 would have sufficed. It is invisible in dev (small N, local DB) and dominant in production.

PatternQueries for 200 tickets
Loop reading t.order.order_code201 — one list + one per row
select_related('order') then the same loop1 — a single JOIN

2.3select_related

An ORM directive: JOIN the named single-valued relations (ForeignKey, OneToOneField) into the same query, so later attribute access is free.

for t in Ticket.objects.all():                       # 1 query
    print(t.order.order_code)                        # +1 query PER ticket → N+1

for t in Ticket.objects.select_related('order'):   # 1 query, JOINed
    print(t.order.order_code)                        # free — already in memory

2.4The limit of select_related

A JOIN can only be planned when the target table is known while the SQL is being built. Django ships one relation whose target table varies per row — the generic foreign key, the subject of Chapter 3. It cannot be select_related: every access is a fresh query per row, and the fix is a manual batch (§3.6).

Chapter 03

Generic relations: ContentType & GFK

The goal: one attachment table that points at Order today and Refund tomorrow — with no new columns and no new join tables.

3.1The ContentType framework

django.contrib.contenttypes ships a built-in table holding one row per model in the project, keyed (app_label, model) — so ('store', 'order') uniquely names the Order model. Django creates the rows automatically after each migrate. Effect: "which model?" becomes data — a value a column can store.

3.2GenericForeignKey (GFK)

Three parts acting as a typed union:

  • content_type — FK to ContentType: which kind of row?
  • object_id — a plain value column: which row of that kind?
  • content_object — a virtual accessor resolving both into the actual instance. No column of its own.
class DocumentAttachment(models.Model):
    document       = models.ForeignKey(Document, related_name='attachments',
                                       on_delete=models.CASCADE)
    content_type   = models.ForeignKey(ContentType, on_delete=models.PROTECT,
                                       limit_choices_to=allowed_targets_q)
    object_id      = models.UUIDField()          # plain value — NO FK constraint
    content_object = GenericForeignKey('content_type', 'object_id')  # virtual

Without the GFK, attaching documents to a fourth entity type would mean a fourth nullable FK column plus a check constraint enforcing "exactly one set." With it, the schema never changes as types arrive.

Gotcha — no cascade

object_id carries no FK constraint, so the database does not know the link exists. Deleting a target row leaves attachment rows dangling — cleanup must happen in code. Chapter 6 centralizes it.

3.3The attachment registry pattern

Which models may receive attachments is a plain Python list of (app_label, model) pairs — the single authority every other piece reads:

ALLOWED_ATTACHMENT_TARGETS: list[tuple[str, str]] = [
    ('store', 'order'),
    ('store', 'invoice'),
    ('store', 'refund'),   # new type = one line, NO migration
]

def resolve_attachment_target(entity_type: str) -> ContentType:
    for app_label, model in ALLOWED_ATTACHMENT_TARGETS:
        if model == entity_type:
            return ContentType.objects.get_by_natural_key(app_label, model)
    raise UnknownTargetError(entity_type)

Why no migration is needed:

  • The content_type FK column already exists on DocumentAttachment.
  • The new model's ContentType row already exists — Django created it at migrate time.
  • object_id is a plain UUID column and every model uses UUID primary keys — the type already fits.

3.4ContentType.objects.get_for_model(M)

Model class in, its ContentType row out — the standard way to filter attachment rows by target type. Lookups are cached in a process-local dict, but the first call per model per process hits the DB.

Tip

In a loop, hoist get_for_model out — resolve the ContentType once, then filter with it per iteration.

3.5limit_choices_to — admin-only by default

The limit_choices_to=allowed_targets_q on the FK constrains which ContentType rows appear as choices in forms and the admin. It is evaluated at query time — not baked into the schema — and it enforces nothing at the database or API layer.

Warning

Belt and suspenders: the single seam (§1.2) must re-validate targets in code — that is exactly what resolve_attachment_target inside _resolve_or_raise does. limit_choices_to alone protects only the admin UI.

3.6GFK and the N+1 — the batch fix

As promised in §2.4: each attachment.content_object access is a fresh query, and GFKs cannot be auto-select_related because the target table differs per row. The fix: group ids by type, run one query per type, patch the results back on.

# one page of attachments → group ids by type → ONE query per type
ids_by_ct = defaultdict(set)
for att in page:
    ids_by_ct[att.content_type_id].add(att.object_id)

targets = {}
for ct_id, ids in ids_by_ct.items():
    Model = ContentType.objects.get_for_id(ct_id).model_class()
    for obj in Model.objects.select_related('order').filter(pk__in=ids):
        targets[(ct_id, obj.pk)] = obj

for att in page:                    # patch — no per-row query
    att._target = targets.get((att.content_type_id, att.object_id))
Warning

Add a fifth attachable type and forget to extend the batch helper: nothing breaks, list performance silently degrades back to N+1. Treat the registry and the batch helper as a pair.

Chapter 04

Files & storage

The goal: the same code writes bytes to local disk in dev and to object storage in prod, with zero branching at call sites.

4.1Storage backends — the pluggable bytes layer

A model's FileField never touches bytes itself; it delegates to a Storage instance. Default is FileSystemStorage rooted at MEDIA_ROOT; S3Boto3Storage (from django-storages) swaps in object storage transparently. The backend is selected once, at boot:

def get_document_storage() -> Storage:
    if settings.DOCS_S3_BUCKET:               # decided once, at app boot
        return S3Boto3Storage(bucket_name=settings.DOCS_S3_BUCKET,
                              file_overwrite=False)
    return FileSystemStorage(location=MEDIA_ROOT / 'documents')

class Document(models.Model):
    file = models.FileField(upload_to=date_partitioned_path,
                            storage=get_document_storage)  # callable → lazy

Every caller — API upload, backfill command, test — writes through doc.file.save(name, content) and never knows which backend answered. No if S3 else FS exists outside the factory.

4.2The file-wrapper family — pick the right one

FileField expects a file-like with Django's attributes (name, size, read(), …). Three wrappers provide them:

You haveWrap it inTypical place
A real file on diskFile(fh, name=…)Backfills, imports
Bytes in memoryContentFile(b'…')Generated files
A fake upload for a testSimpleUploadedFile(name, bytes, content_type)Testing upload endpoints

Real multipart uploads arrive already wrapped — Django hands views an InMemoryUploadedFile or TemporaryUploadedFile. The wrappers above are for everything that isn't an HTTP upload:

with open(pdf_path, 'rb') as fh:
    upload = File(fh, name=pdf_path.name)      # disk file → FileField-ready
    document_service.create_with_attachments(uploaded_file=upload, ...)
Tip

Open file handles lazily, inside the loop, with with. Opening a 50k-file folder up front exhausts file descriptors; the context manager closes each handle as soon as its storage write completes.

4.3storage.exists(name) — backend-agnostic checks

Works identically on FileSystemStorage and S3Boto3Storage. In tests, it asserts the blob write actually happened, not merely that DB rows exist:

fake = SimpleUploadedFile('note.pdf', b'%PDF-1.4 ...',
                          content_type='application/pdf')
doc = document_service.create_with_attachments(uploaded_file=fake, ...)
self.assertTrue(doc.file.storage.exists(doc.file.name))  # FS and S3 alike
Warning — recall §1.3

Storage writes are not transactional. Django's TestCase rolls back DB rows between tests, but blobs persist — point MEDIA_ROOT at a temp dir (override_settings) and clean it up, or tests will leak files into each other.

Chapter 05

Commands & data migrations

Two vehicles for one-shot operational work: management commands (run by an operator, any time) and data migrations (run by migrate, exactly once per environment).

5.1Management commands — discovery by path

Any module at <app>/management/commands/<name>.py defining a class named Command(BaseCommand) becomes a manage.py subcommand. Auto-discovered — no registration.

Why a command and not a standalone script:

  • Django context comes free — settings, app registry, DB connections; no manual django.setup().
  • Discoverable — it appears in manage.py help next to its siblings.
  • Testable — invocable in-process from tests (§5.3).

5.2BaseCommand anatomy + CommandError

Override two methods: add_arguments(parser) registers argparse flags; handle(...) does the work, receiving the parsed flags as keyword args (names derived from the long flag, - → _). Raise CommandError to exit with code 1 and a clean stderr message — no traceback noise.

# store/management/commands/backfill_invoice_pdfs.py
class Command(BaseCommand):
    help = 'Attach invoice PDFs from a local folder to their Orders'

    def add_arguments(self, parser):
        parser.add_argument('--source-dir', required=True, type=Path)
        parser.add_argument('--apply', action='store_true')  # dry-run by default

    def handle(self, *, source_dir, apply, **opts):   # **opts mops Django's own kwargs
        if not source_dir.is_dir():
            raise CommandError(f'{source_dir} is not a directory')  # exit 1, clean stderr
        for pdf in sorted(source_dir.glob('*.pdf')):
            order = self._resolve(pdf.name[:4])       # the Substr lookup from §2.1
            if order is None or order.invoice_pdf_id:
                continue                              # idempotent: re-runs skip
            if apply:
                with transaction.atomic():            # per-iteration granularity (§1.3)
                    with open(pdf, 'rb') as fh:
                        document_service.create_with_attachments(
                            uploaded_file=File(fh, name=pdf.name), ...)
        self.stdout.write(self.style.SUCCESS('done'))

Note the capstone: per-iteration atomic() (§1.3), the single seam (§1.2), the File wrapper (§4.2) — one command exercising every prior chapter.

Tip — one-shot command hygiene

Dry-run by default, --apply commits. Make re-runs skip already-processed rows instead of adding a --force flag — idempotence beats options. Catch only the exceptions you expect per item (I/O, transport, constraint violations); let programming bugs crash on the first item, not drown as N error rows.

5.3call_command — the test entry point

Programmatic, in-process invocation: no subprocess, no auth. Capture output through StringIO:

out = StringIO()
call_command('backfill_invoice_pdfs',
             '--source-dir', tmp_dir, '--apply', stdout=out)
self.assertIn('done', out.getvalue())

5.4Data migrations — RunPython

migrations.RunPython(forward, reverse) is a migration operation that runs Python instead of DDL. The forward function receives (apps, schema_editor) where apps is a registry pinned to that migration's snapshot of model state — the schema as it was at that point in history, not today's classes.

def prune_orphans(apps, schema_editor):
    Attachment = apps.get_model('store', 'DocumentAttachment')  # historical snapshot
    # … delete rows whose (content_type, object_id) matches no live target …

class Migration(migrations.Migration):
    operations = [
        migrations.RunPython(prune_orphans, migrations.RunPython.noop),
    ]
  • No-op reverse. When forward removes data that cannot be restored, pass RunPython.noop as the reverse. It keeps migrate store 00NN (rolling back past this migration) runnable without errors, instead of raising.
  • Make it idempotent. On a fresh DB or an already-clean DB the function should find nothing to do.
Gotcha — snapshots are not your classes

Snapshot classes are plain reconstructions: custom model methods are absent, and any hook registered against your live classes — including the lifecycle signals of the next chapter — never fires inside RunPython. Convenient when pruning (no recursive re-triggering); confusing if you expected cleanup logic to run.

Chapter 06

Signals

The dangling problem from §3.2 — object_id has no FK, so the DB cannot cascade — meets its real difficulty here: deletes come from everywhere. Instance .delete(), queryset .delete(), FK cascade collecting children, admin endpoints, the shell. Duplicating cleanup in each path drifts.

6.1Model lifecycle signals (pre_delete & co.)

Django fires pre_delete for every row about to be deleted, regardless of trigger. A receiver connected to the signal becomes a property of the model itself, not something each delete path must remember. Siblings: post_delete, pre_save, post_save.

Tip — cascades are covered

Django's deletion Collector walks the FK graph first, and fires pre_delete for each row it intends to remove — then runs the bulk DELETE. Deleting an Order fires the signal on its collected Ticket children too. You never walk the cascade yourself.

6.2One receiver, driven by the registry

# store/signals.py
def _detach_documents_on_delete(sender, instance, **_):
    ct = ContentType.objects.get_for_model(sender)          # §3.4
    DocumentAttachment.objects.filter(
        content_type=ct, object_id=instance.pk,
    ).delete()                       # link severed; Document rows + blobs survive

def connect_attachment_pre_delete_signals():
    for app_label, model_name in ALLOWED_ATTACHMENT_TARGETS:  # §3.3
        Model = apps.get_model(app_label, model_name)
        pre_delete.connect(_detach_documents_on_delete, sender=Model,
                           dispatch_uid=f'detach_documents_{model_name}')

The wiring loop walks the same registry that drives upload validation — a type added to the list later inherits the detach behavior with zero extra wiring.

6.3AppConfig.ready() — the canonical wiring hook

Runs once per process, after the app registry has settled — the sanctioned place to connect signals:

# store/apps.py
class StoreConfig(AppConfig):
    def ready(self):                          # once per process
        from store.signals import connect_attachment_pre_delete_signals
        connect_attachment_pre_delete_signals()

The import is deliberately inside ready(): a module-top import of code touching other apps' models risks AppRegistryNotReady in some startup orderings.

6.4dispatch_uid — idempotent wiring

The de-dup key on Signal.connect: the same uid + sender registers the receiver once, however many times the connect call runs (test setup, the dev autoreloader, repeated imports). Without it, duplicate registrations mean the handler fires twice per event.

6.5apps.get_model(label, name) — lazy lookup, two registries

Looks a model up by string, deferring class resolution to registry-ready time — which is why the wiring loop can name models it must not import at module top.

Gotcha — same name, different registry

In normal code, django.apps.apps.get_model(...) returns your live class. Inside RunPython (§5.4), the apps argument is the historical snapshot registry — same method name, different classes. And because signals are connected to live classes, migrations never fire them.

Trade-off — signals are invisible

A maintainer reading a delete view sees no detach logic; it lives in another file. While the surface is small, an explicit service call (e.g. delete_refund() that detaches then deletes) beats clever. Switch to the signal when delete paths multiply — several admin endpoints × cascades × future paths — and the visible-but-duplicated approach starts to drift.
