1 Object storage & the S3 API

Start here — everything below builds on these three words: bucket, object, key.

Object storage keeps each file as a self-contained object (its bytes + metadata) inside a bucket (a container). You address an object by its key — its name within the bucket. There are no real folders: a key like invoices/42/report.pdf is one flat string; the slashes are just convention (a "prefix" you can filter by).

S3 (Amazon Simple Storage Service) is AWS's object store. Its HTTP API became the de-facto industry standard, so "speaks S3" means a tool can talk to any store that implements that same request/response contract — using the same SDKs (like Python's boto3).

Keep in mindAn object is written and read whole, by key. That's why file uploads are a perfect fit — upload once, download whole, delete by key. No partial edits.

2 What MinIO is

MinIO is a high-performance, S3-compatible object store you can run yourself. The community edition is open source under the GNU AGPLv3 license. Because it implements the S3 API, any S3 SDK, CLI, or library works against it — the same code you'd point at AWS.

Operationally it's a single server process that stores objects under a data directory on disk. That's all you need locally: one container, one data volume, speaking S3.

The one idea that mattersMinIO is not "like" S3 with a different API — it is the S3 API, served by a program you control. Swapping AWS S3 for MinIO is a change of address, not of code.

3 Why you'd use it — a worked example

Take a web app with a user file-uploads feature: users upload documents and the app stores them in an S3 bucket. In dev and prod that's a real AWS bucket (myapp-<env>-uploads). But the local Docker stack has no AWS — and you don't want a separate "local filesystem" code path that never runs in production and quietly drifts.

MinIO solves that: it runs as a container in the local stack and speaks the same S3 API. So the backend's storage code — a boto3 S3 client — runs unchanged everywhere. Only the endpoint differs:

EnvironmentS3 endpointCredentialsBucket
Localhttp://minio:9000 (MinIO)minioadmin / minioadminuser-uploads
Dev / ProdAWS S3 (default endpoint)IAM role (e.g. ECS task-role — no keys in env)myapp-<env>-uploads
PayoffOne code path, exercised locally exactly as in prod. No untested fallback branch, no "works on my machine" storage bugs.

4 Running the MinIO server

MinIO starts with the minio server command pointed at a data directory. The local stack runs exactly:

minio server /data --console-address ":9001"
  • /data — the directory where objects are stored (mount a Docker volume there so files survive restarts).
  • MinIO exposes two interfaces on two ports:
PortInterfaceWho uses it
9000S3 APIApplications / SDKs (the app's boto3 client)
9001Web ConsoleHumans, in a browser (pinned by --console-address ":9001")

Root credentials

MinIO's initial admin user comes from two environment variables, MINIO_ROOT_USER and MINIO_ROOT_PASSWORD, each defaulting to minioadmin. Locally the defaults are fine; the console at http://localhost:9001 logs in with minioadmin / minioadmin.

WarningThe docs are explicit: never use the default credentials outside local development. In this setup that's not an issue in dev/prod — there the app uses real AWS S3, not MinIO, so there are no MinIO root credentials to leak. MinIO is a local-only dependency.

5 Buckets & the MinIO Client (mc)

A fresh MinIO server has no buckets — nothing auto-creates them. You make them yourself, with the MinIO Client (mc), a command-line tool for S3-compatible stores. Two commands matter here:

CommandDoes
mc alias set NAME URL ACCESS SECRETRegisters a target server under a short alias, with credentials, so later commands can just say NAME/....
mc mb ALIAS/BUCKET"Make bucket" — creates a bucket on that target. Flags: --ignore-existing (no error if it exists), --with-versioning, --with-lock.

The stack does this once, at startup, in a throwaway minio-setup container:

# register the local server as alias "local", then create the bucket
mc alias set local http://minio:9000 minioadmin minioadmin
mc mb -p local/user-uploads
Why a separate setup stepThe server (§4) and bucket creation are two jobs. Keeping minio-setup as its own one-shot container — it runs, makes the bucket, exits — mirrors the familiar one-shot DB-init container pattern, and the app waits for it before starting.

mc — MinIO Client · mc mb — Make Bucket

6 Talking S3 from application code

The backend uses boto3 (the AWS SDK for Python). Two settings make it talk to MinIO instead of AWS:

  1. Endpoint — set endpoint_url to the MinIO URL (http://minio:9000). Leave it unset in dev/prod so boto3 uses the real AWS S3 endpoint.
  2. Credentials — the access/secret keys (locally minioadmin; in dev/prod, an IAM role — e.g. an ECS task-role — supplies them automatically).

One more setting is needed, and it's worth understanding: path-style addressing.

Path-style vs virtual-hosted-style

S3 has two ways to put the bucket into the request URL:

StyleURL shapeBucket goes in the…
Virtual-hosted (AWS default)https://bucket.s3.region.amazonaws.com/keyhostname (subdomain)
Path-stylehttps://s3.region.amazonaws.com/bucket/keyURL path

Virtual-hosted style needs every bucket to be a working DNS name. Against a custom endpoint like minio:9000 that fails — user-uploads.minio:9000 resolves to nothing. Path-style keeps the host fixed and puts the bucket in the path, so it works against any endpoint. In boto3 that's:

boto3.client(
    "s3",
    endpoint_url=settings.AWS_S3_ENDPOINT_URL or None,  # http://minio:9000 locally; empty → real AWS
    config=Config(s3={"addressing_style": "path"}),
)
GotchaForget addressing_style: "path" and boto3 defaults to virtual-hosted — it will try bucket.minio:9000, which has no DNS, and every call fails with a connection error. This is the single most common MinIO-from-SDK mistake.

8 Gotchas & cheat-sheet

Path-style is mandatoryCustom endpoint → set addressing_style: "path", or DNS-based bucket hostnames break every call.
Buckets aren't automaticMinIO starts empty. If the app 404s on the bucket, the minio-setup / mc mb step didn't run or didn't finish before the app.
Endpoint must be empty in dev/prodIf AWS_S3_ENDPOINT_URL is accidentally set in a cloud env, boto3 points at a non-existent local host instead of AWS S3.
9000 vs 9001The app talks to 9000 (S3 API). 9001 is the human Console — never point the SDK there.
