01

What React is — and where it sits next to Django

React describes itself as "the library for web and native user interfaces". It is a JavaScript library — not a framework — for building user interfaces out of individual pieces called components.

The mental shift from Django templates is where and when HTML gets produced:

  • Django: the server renders a template once per request and sends finished HTML. Any change means a new request.
  • React: JavaScript runs in the browser, builds the page, and keeps it alive. When data changes, React re-runs the affected components and patches only the parts of the page that differ — no reload.
You know (Django)React counterpartKey difference
Template (.html)Component (a function)Re-runs in the browser whenever its data changes
Template context dictPropsRead-only inputs, passed per component
{% include %} with argsRendering a child componentComposition is the primary structuring tool
New request/responseRe-renderNo page reload; React updates the live page in place
DB/session as source of truthState (in-memory)Lives in the browser tab; refresh resets it
Tip

The official docs live at react.dev and are excellent — everything in this guide is grounded in them. Older content at reactjs.org or tutorials using class MyComponent extends React.Component describe the legacy style; modern React (and everything below) is function components + hooks.

02

Components: functions that return markup

The Quick Start puts it in one line: "React components are JavaScript functions that return markup." A component is the unit of UI — a button, a table row, a whole page. Apps are trees of components rendering other components. 

function ProductCard() {
  return (
    <article>
      <h3>Walnut Desk</h3>
      <p>45 in stock</p>
    </article>
  );
}

export default function ProductList() {
  return (
    <section>
      <ProductCard />
      <ProductCard />
    </section>
  );
}
  • Capitalization is load-bearing. Component names must start with a capital letter. Lowercase tags (<section>) are treated as HTML; capitalized tags (<ProductCard />) are your components.
  • One component per concern; files typically export one main component (export default).
Gotcha

Components can render other components, but never nest their definitions — defining function Profile() {…} inside another component's body is, per the docs, "very slow and causes bugs". Declare every component at the top level of the file; when a child needs data from a parent, pass it by props (§5).

03

JSX: markup inside JavaScript

The HTML-looking syntax those functions return is JSX. It looks like HTML but is stricter, and it lets you interleave live JavaScript. Three rules:

  1. Return a single root element. To group siblings without adding a wrapper node, use a Fragment: <> … </>.
  2. Close every tag — including self-closing ones: <img />, <br />.
  3. camelCase most attributes. class is a reserved word in JavaScript, so it becomes className; onclick becomes onClick. Exception: aria-* and data-* attributes keep their dashes.

Curly braces — the window into JavaScript

Inside JSX, { } escapes back into JavaScript: any expression works. Braces are valid in exactly two positions — as text content inside a tag, or as an attribute value right after =. 

const product = { name: 'Walnut Desk', stock: 45 };

export default function ProductCard() {
  return (
    <article className="card" style={{ padding: 16 }}>
      <h3>{product.name}</h3>
      <p>{product.stock} in stock · {product.stock * 2} sold</p>
    </article>
  );
}
Tip

style={{ padding: 16 }} is not special syntax — it's an ordinary JS object literal { padding: 16 } sitting inside the JSX braces. Whenever you see {{ }}, read it as "an object passed as a value".

Warning

Braces take expressions, not statements — no if or for directly inside JSX. Put statements above the return, or use the expression forms from §6 (ternary, &&, .map()).

04

From source file to browser: the build step

JSX is not standard JavaScript — a browser can't execute a .jsx file directly. A build tool compiles it to plain JS, packages the source, serves it during development, and produces the deployable bundle. The official docs recommend Vite, Parcel, or Rsbuild for exactly this job: packaging source code, a local dev server, and a production build command.

In a Vite project (like a typical Django-API + React setup) the loop is:

CommandWhat happens
pnpm devDev server with Hot Module Replacement (HMR) over native ES modules — edits show up in the browser near-instantly, usually keeping app state.
pnpm buildProduction build: bundles the code (Vite's current bundler is Rolldown) into static JS/CSS you deploy behind any web server.
  • Files are .jsx — or .tsx when TypeScript is in play (§15).
  • Compile errors surface in the Vite terminal and as an in-browser overlay — the FE equivalent of a Django traceback page.
  • The browser never sees your source tree; it sees the compiled output. When debugging, the browser DevTools "Sources" tab maps back via source maps.
05

Props: read-only inputs

Props are how a parent passes data to a child — the closest analogue to a Django template's context dict. JSX attributes become a single props object; components conventionally destructure it in the signature. 

function ProductCard({ name, stock, category = 'Furniture' }) {
  return (
    <article>
      <h3>{name}</h3>
      <p>{stock} in stock · {category}</p>
    </article>
  );
}

// Usage — each attribute becomes a prop
<ProductCard name="Walnut Desk" stock={45} />
  • Props are immutable — "read-only snapshots in time" in the docs' words. A component must never change its own props; if it needs different data, it must ask its parent to pass different props (how a child "asks" is §7 + §11).
  • Default values go in the destructuring: category = 'Furniture'.
  • Strings can be plain attributes; everything else (numbers, objects, functions) goes through braces: stock={45}.

The children prop

JSX nested between a component's tags arrives as a special prop called children — this is how wrapper/layout components work:

function Panel({ title, children }) {
  return (
    <section className="panel">
      <h2>{title}</h2>
      {children}
    </section>
  );
}

<Panel title="Inventory">
  <p>Any JSX here becomes {'children'}.</p>
</Panel>
06

Conditional rendering and lists

There is no {% if %} / {% for %} — control flow is plain JavaScript:

Django templateReact / JSX
{% if %} … {% else %}if above the return, or cond ? a : b inline
{% if %} (no else)cond && <Thing />
{% for %}items.map(item => <Row key={item.id} />)
 
function StatusBadge({ status }) {
  if (status == null) {
    return null;               // renders nothing at all
  }
  return <span>{status}</span>;
}

// Inline forms
<p>{isConfirmed ? 'Confirmed' : 'Draft'}</p>
{unreadCount > 0 && <Badge count={unreadCount} />}

React treats false, null, and undefined as "holes" — nothing is rendered for them.

Gotcha

Don't put numbers on the left side of &&. {unreadCount && <Badge />} renders a literal 0 when the count is zero (0 is falsy but still renderable). Always compare: {unreadCount > 0 && <Badge />}.

Lists and the key prop

export default function OrderList({ orders }) {
  return (
    <ul>
      {orders.map(o => (
        <li key={o.id}>{o.customerName}</li>
      ))}
    </ul>
  );
}

key tells React which rendered item corresponds to which data item across re-renders. Keys must be unique among siblings and must not change — use a stable ID from your data (a DB primary key or UUID from the Django API is ideal).

Warning — review flags

key={index} "often leads to subtle and confusing bugs" (docs' words) once items are inserted, deleted, or reordered — index is also what React silently defaults to. key={Math.random()} is worse: every key changes every render, so React rebuilds all the DOM each time.

07

Events: pass the function, don't call it

Interactivity starts with event handler props — on + capital letter (onClick, onChange, onMouseEnter). You give them a function; React calls it when the event fires. Handlers are conventionally named handle + event. 

function DeliverButton({ onDeliver }) {
  function handleClick() {
    onDeliver();
  }
  return <button onClick={handleClick}>Deliver</button>;
}
Gotcha

onClick={handleClick} passes the function. onClick={handleClick()} calls it immediately, during rendering — a classic bug. Inline arrows are the correct way to pass arguments: onClick={() => onSelect(product.id)}.

Note the pattern above: onDeliver is just a prop that happens to hold a function. Parents hand behavior to children the same way they hand data — this becomes the backbone of "data down, events up" in §11.

08

State: a component's memory

A local variable can't power interactivity, for two reasons the docs call out: local variables don't persist between renders, and changing them doesn't trigger a render. State fixes both — it's memory that survives re-renders, and updating it tells React to re-render. 

import { useState } from 'react';

export default function PriceKnob() {
  const [factor, setFactor] = useState(1);   // [value, setter], initial 1

  return (
    <>
      <button onClick={() => setFactor(factor - 0.05)}>–</button>
      <span>{factor.toFixed(2)}</span>
      <button onClick={() => setFactor(factor + 0.05)}>+</button>
    </>
  );
}
  • useState returns an array: the current value and a setter. Calling the setter re-renders the component with the new value.
  • useState is a Hook — a function whose name starts with use, giving a component access to React features.
  • Rules of Hooks: call them only at the top level of a component (or another Hook) — never inside conditions, loops, or nested functions. Think of them as unconditional declarations of what the component needs.
  • State is isolated per instance. Render <PriceKnob /> twice and each copy has its own independent factor. It's also fully private — a parent can't reach in and change it.

The render cycle: trigger → render → commit

Before UI reaches the screen, React always runs the same three steps:

Three consequences of this model explain most "React is being weird" moments:

State is a snapshot

Each render sees a fixed value of its state. Setting state doesn't change the variable you already have — it requests a new render, which will see the new value. This holds even in async code inside the same handler. 

function handleClick() {
  setFactor(factor + 0.05);
  console.log(factor);   // still the OLD value — this render's snapshot
}

Updates are batched

React waits until all code in an event handler has run before re-rendering. Setting the same state twice from the same snapshot doesn't stack; to update based on the pending value, pass an updater function: 

setCount(count + 1);
setCount(count + 1);     // same snapshot → net effect +1

setCount(c => c + 1);
setCount(c => c + 1);    // updater functions → +2

Rendering must be pure

React assumes every component is a pure function: same props/state in, same JSX out, and no changing of pre-existing variables or objects during render. Side effects belong in event handlers (§7) — or, as a last resort, Effects (§12).

Tip

Seeing components run twice in development (double logs, double fetches)? That's Strict Mode: React deliberately calls each component's function twice in dev to surface impure components. It's a feature, not a bug — don't "fix" it by removing <StrictMode>.

10

Updating objects and arrays: replace, never mutate

Purity (§9) has a practical corollary: treat any object you put into state as read-only. Mutating it in place changes data React has no way to notice — the UI won't reliably update. Instead, build a new object and hand it to the setter. 

// ❌ Mutation — React can't see this change
series.july.total = 210;
setSeries(series);

// ✅ Replace — new object at every level you touch
setSeries({
  ...series,
  july: { ...series.july, total: 210 },
});

The same rule covers arrays (they're objects too): produce new ones with [...rows, newRow], rows.map(…), rows.filter(…) — not push/splice on the array already in state.

Tip

Local mutation is fine. Mutating an object you just created in the same handler — before anything else references it — is explicitly OK. The rule protects objects that are already in state, not fresh ones you're still assembling.

11

Sharing state: lift it up

When two components must stay in sync, neither should own the data. The pattern — lifting state up — is: remove the state from both, move it to their closest common parent, and pass it back down via props. Each piece of state then has exactly one owning component: the single source of truth. 

export default function App() {
  const [selectedId, setSelectedId] = useState(null);

  return (
    <>
      <ProductList selectedId={selectedId} onSelect={setSelectedId} />
      <DetailPanel productId={selectedId} />
    </>
  );
}

The docs frame this as controlled vs uncontrolled: a component driven by props (parent decides) is "controlled"; one keeping its own local state is "uncontrolled". Lifting state converts the former into the latter's coordinated version.

12

Effects: synchronizing with the outside world

Everything so far stays inside React. An Effect (useEffect) is the escape hatch for code that must talk to an external system — the network, browser APIs, non-React widgets. For a backend developer, the headline use case: fetching from your Django API. 

import { useState, useEffect } from 'react';

function OrderDetail({ id }) {
  const [order, setOrder] = useState(null);

  useEffect(() => {
    let ignore = false;

    fetch(`/api/orders/${id}/`)
      .then(res => res.json())
      .then(data => {
        if (!ignore) setOrder(data);
      });

    return () => {
      ignore = true;   // cleanup: stale responses for old ids get discarded
    };
  }, [id]);              // re-run whenever id changes

  if (order == null) return <p>Loading…</p>;
  return <h2>{order.customerName}</h2>;
}

The dependency array

FormWhen the Effect runs
useEffect(fn)After every render
useEffect(fn, [])Only on mount (first appearance on screen)
useEffect(fn, [a, b])On mount, and whenever a or b changed since the last render
  • Cleanup: the function you return runs before the Effect runs again, and one final time on unmount. Use it to cancel, unsubscribe, or (as above) flag stale async results.
  • Dev double-run: in development, Strict Mode remounts every component once right after mount — so effect + cleanup + effect. It's a stress test for missing cleanup; the docs explicitly warn against suppressing it (e.g. with refs).
Gotcha

A fetch Effect without cleanup has a race: navigate from order A to B quickly, and A's slower response can land last and overwrite B's data. The ignore flag above is the docs' own fix.

13

You might not need an Effect

The most common React code smell — and the docs dedicate a page to it. Two cases where an Effect is the wrong tool:

  • Transforming data for rendering. Don't mirror derived values into state and sync them with an Effect — compute them during render.
  • Handling user events. Code that responds to an action (a click sent a POST) belongs in the event handler, where you know exactly what happened — not in an Effect watching for side-channel state changes.
// ❌ Derived state synced via an Effect — extra render pass, drift risk
const [total, setTotal] = useState(0);
useEffect(() => {
  setTotal(price * quantity);
}, [price, quantity]);

// ✅ Just derive it during render
const total = price * quantity;

The docs' promise: removing unnecessary Effects makes code "easier to follow, faster to run, and less error-prone".

Review heuristic

For every useEffect in a diff, ask: which external system does this synchronize with? If the answer is "none — it computes something from props/state" or "it reacts to a user action", it should be a plain expression or live in the handler.

14

The wider hooks toolbox

useState and useEffect cover most code you'll read. The official reference groups the rest by purpose:

CategoryHooksOne-liner
StateuseState, useReducerComponent memory; useReducer centralizes update logic in a reducer function
ContextuseContextRead values provided by an ancestor without prop-passing
RefsuseRefHold a value that isn't needed for rendering (often a DOM node)
EffectsuseEffectSynchronize with external systems (§12)
PerformanceuseMemo, useCallbackCache a computed value / function identity between renders to skip redundant work

useRef — mutable, silent

A ref persists across renders like state, but is mutable and changing ref.current does not trigger a re-render. Classic uses: holding a DOM node, timer IDs, "previous value" bookkeeping. 

const inputRef = useRef(null);

<input ref={inputRef} />
<button onClick={() => inputRef.current.focus()}>Focus</button>
Warning

Do not read or write ref.current during rendering (except one-time initialization) — only from event handlers or Effects. During render it breaks the purity contract from §9.

Context — skipping the prop relay

Context lets a parent make a value available to any component below it, however deep — the alternative to "prop drilling" the same prop through many layers. Three parts: create, provide, read. 

const ThemeContext = createContext('light');

function App() {
  return (
    <ThemeContext value="dark">
      <Toolbar />
    </ThemeContext>
  );
}

function DeliverButton() {          // anywhere under Toolbar, any depth
  const theme = useContext(ThemeContext);
  // ...
}

The current docs render the context itself as the provider (<ThemeContext value="dark">); in existing codebases you'll usually meet the older equivalent, <ThemeContext.Provider value="dark">.

The docs' own advice before reaching for it: first try just passing props, or passing components as children. Context is for genuinely tree-wide concerns (theme, current user, locale).

Custom hooks — extracting stateful logic

Any function named use + Capital that calls other hooks is a custom hook. They share stateful logic, not state itself — every call gets fully independent state. The §12 fetch pattern extracts naturally: 

function useOrder(id) {
  const [order, setOrder] = useState(null);
  useEffect(() => {
    // fetch + ignore-flag cleanup, exactly as in §12
  }, [id]);
  return order;
}

// Any component: const order = useOrder(id);
15

TypeScript: typed props in .tsx

Three mechanics cover most of what you'll see in a TS React codebase:

  • Every file containing JSX must use the .tsx extension.
  • Type definitions come from @types/react and @types/react-dom (dev dependencies).
  • Props are typed with an interface (or type) annotated on the destructured parameter.
interface ProductCardProps {
  name: string;
  stock: number;
  onSelect?: (id: string) => void;   // ? = optional prop
}

function ProductCard({ name, stock, onSelect }: ProductCardProps) {
  // ...
}
Tip

When reviewing: the props interface is the component's contract — the same role a DRF serializer plays for an endpoint. Reading it first tells you what the component consumes before you read a line of its body.

16

Thinking in React: the official 5-step method

The docs' canonical process for going from mockup to app — equally useful as a rubric when reviewing a component design:

  1. Break the UI into a component hierarchy — draw boxes around every piece; one component ≈ one responsibility.
  2. Build a static version — all props, no state; render the data without interactivity first.
  3. Find the minimal but complete representation of UI state — anything computable from props or other state (a filtered list, a count, a line total from price × quantity) should be derived, not stored.
  4. Identify where state should live — the closest common parent of every component that needs it (§11).
  5. Add inverse data flow — pass setter callbacks down so children can request changes (§7 + §11).
Review heuristic

Steps 3 and 4 catch the majority of design problems in PRs: state that duplicates other state, and state living in the wrong component. Ask "could this be derived?" and "who else needs this?" before anything else.
