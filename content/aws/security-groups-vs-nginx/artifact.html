Security Groups vs NGINX — Layers of Defense

How AWS Security Groups (a network-layer firewall) and NGINX (an application-layer reverse proxy) sit in series along the request path and enforce different kinds of access control.

Network defense · how the layers fit

Security Groups & NGINX: two gates, two layers

A Security Group and NGINX are not alternatives — a request passes through both, at different layers, in a fixed order. Understanding which gate does what (and what neither can do for the other) is the key to locking a service down correctly.

Why this matters. Any team's "VPN-only" policy lives at exactly these two layers — Security Group IP rules and NGINX allow/deny blocks (plus the ALB's equivalent on the ECS path). A common misconfiguration — raw ports bypassing nginx — is a textbook case of the two getting out of sync. Read top-to-bottom; concepts build in order.

01 The one idea: layers in series

A Security Group (SG) and NGINX answer two different questions about the same request, and they answer them one after the other:

  • Security Group — "Is this source IP even allowed to open a connection to this port on this machine?" (network/transport layer)
  • NGINX — "Now that an HTTP request has arrived on a port I'm serving, what do I do with it — which backend, allowed or denied, decrypt TLS, require auth?" (application layer)

Both must say "yes" for traffic to reach your app. That's defense in depth: two independent gates, so a mistake at one layer doesn't automatically expose everything.

The SG is enforced by the AWS network fabric before the packet ever reaches the instance's OS. Only if the SG admits it does the packet hit the host port where NGINX is listening. (There's also a stateless subnet-level NACL even earlier, before the SG — out of scope here; SGs are the layer you'll actually tune.)

03 Security Group — the edge firewall

A Security Group is a stateful virtual firewall attached to an instance's network interface (ENI). It's the outermost gate you manage, and it works purely at the network/transport layer:

  • It sees: source (an IP/CIDR, another security group, or a prefix list), protocol (TCP/UDP/ICMP), and port. Nothing above that.
  • It's blind to: the HTTP request — URL paths, hostnames, headers, cookies — and to TLS content (it can't decrypt; it only sees "TCP to port 443").
  • Allow-only: rules can only permit traffic. Anything not explicitly allowed is denied — there's no "deny" rule.
  • Stateful: if inbound is allowed, the return traffic is automatically allowed.

Its single decision: "can this source reach this port at all?" That makes it perfect for coarse rules like "port 9443 only from the VPN" — and useless for anything path-aware.

Position

At the VPC/ENI boundary, in front of the host. A packet blocked by the SG never reaches NGINX — or any process on the instance.

04 NGINX — the application gatekeeper

NGINX is a reverse proxy that runs as a process on the host, listening on the ports the SG allows through (typically 80/443). Because it speaks HTTP, it operates at the application layer and sees the whole request:

  • Terminates TLS — decrypts HTTPS, so everything past it can be inspected and routed.
  • Routes by host header and URL path to different upstreams (proxy_pass).
  • Filters per location — allow/deny by client IP, require auth, rate-limit, add security headers.

Its decision: "given an HTTP request that already reached my port, what do I do with it?" This is where fine-grained policy lives — e.g. "/admin is VPN-only but / is public":

nginx per-location rule (typical pattern)

location ^~ /admin { allow 10.8.0.0/24; # VPN clients allow 203.0.113.5; # VPN server deny all; proxy_pass http://app; }

06 Why you need both

Each gate has a gap the other fills:

  • The SG can't do paths. "Public site, but /admin VPN-only" is impossible at the SG — it only sees "port 443." You must use NGINX (or the ALB) for that.
  • NGINX can't protect a port it isn't serving. NGINX only governs traffic to the ports it listens on. If another process binds a different port and the SG opens it, traffic reaches that process directly — NGINX never sees it, and none of its TLS/auth/filtering applies.

So they're complementary: the SG decides which doors exist; NGINX decides what happens once you're through the door it guards. A door the SG opens that NGINX doesn't guard is an unguarded door.

The lesson

NGINX's protections are only as good as the SG's port list. A service on a port NGINX doesn't proxy is reachable to whatever the SG allows. The fix is one of two things: close/restrict that port at the SG, or route the service through NGINX — never leave an open port pointing at an unguarded service.

08 Three places to enforce "VPN-only"

IP allowlisting can happen at any of three layers. Picking the right one depends on granularity and which component sees the real client IP:

The "who sees the real IP" gotcha

Behind a load balancer, the host NGINX sees the balancer's IP, not the client's — so on the ECS path, IP allowlisting must be done at the ALB (or via trusted X-Forwarded-For), not the per-task NGINX. That's why the VPN restriction for ECS services is an ALB source_ip rule, not an nginx deny.
