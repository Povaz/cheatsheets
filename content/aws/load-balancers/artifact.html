01 Why a load balancer

Run your application on a single server and you have two problems: it's a bottleneck (one machine can only handle so much traffic) and a single point of failure (if it dies, your app is down). The fix is to run several copies and put something in front of them to spread the work.

That "something" is a load balancer. In AWS's words, it "automatically distributes your incoming traffic across multiple targets, such as EC2 instances, containers, and IP addresses." Crucially, it also "monitors the health of its registered targets, and routes traffic only to the healthy targets." The payoff:

  • Availability & fault tolerance — if one server fails its health check, traffic shifts to the others automatically.
  • Scalability — add or remove servers behind the load balancer without disrupting the flow of requests.
  • A single front door — clients talk to one stable endpoint, not to individual servers.

Source: What is Elastic Load Balancing? — AWS Docs

02 Elastic Load Balancing — the managed service

On AWS you don't build or run a load balancer yourself. Elastic Load Balancing (ELB) is the managed service that provides one for you. AWS operates it, keeps it highly available, and "scales your load balancer capacity automatically in response to changes in incoming traffic" — you don't provision capacity or patch servers.

Two pieces of AWS vocabulary you need before going further:

  • VPC (Virtual Private Cloud) — your own private virtual network inside AWS. Your load balancer and your servers live in a VPC.
  • Availability Zone (AZ) — one of several isolated data-center locations within an AWS Region. Spreading across AZs is how you survive the failure of a whole data center. ELB is designed to operate across multiple AZs.

Keep these two in mind — they become concrete requirements in §7.

Source: What is Elastic Load Balancing? — AWS Docs

03 Layer 4 vs Layer 7 — the one idea before choosing a type

ELB offers different load balancer types, and the difference between them comes down to how deeply they can look at your traffic. That's described using the OSI networking model. You only need two layers from it:

  • Layer 4 — the transport layer (TCP / UDP). The load balancer sees only IP addresses and ports. It can forward a connection extremely fast, but it has no idea what's inside the traffic. It can't read a URL.
  • Layer 7 — the application layer (HTTP / HTTPS). The load balancer understands the actual web request: the URL path, the host name, headers, cookies. This lets it make content-based routing decisions — e.g. send /api/* to one group of servers and /images/* to another.

Rule of thumb: Layer 7 = smart routing for web apps; Layer 4 = raw speed and protocol-agnostic forwarding. This single distinction explains the entire type menu below.

Sources: Elastic Load Balancing features — AWS · What is an Application Load Balancer? — AWS Docs

04 The four load balancer types

ELB supports four types. Pick by what your traffic is and what you need:

TypeLayerUse it when…
Application Load Balancer (ALB)7 — HTTP/HTTPSYou're running web apps/APIs and want content-based routing (by path, host, headers). The default for most workloads.
Network Load Balancer (NLB)4 — TCP/UDP/TLSYou need extreme performance, very low latency, or a static IP address; or your protocol isn't HTTP.
Gateway Load Balancer (GWLB)3/4You're deploying and scaling third-party virtual appliances (firewalls, intrusion detection, deep packet inspection).
Classic Load Balancer (CLB)4 & 7 (legacy)Don't, for new work. It's the previous generation — AWS recommends migrating off it.

AWS's own guidance: "If you need flexible application management, we recommend that you use an Application Load Balancer. If extreme performance and static IP is needed for your application, we recommend that you use a Network Load Balancer."

From here on: ALB

The rest of this guide focuses on the Application Load Balancer, since it's what you'll reach for first. The core building blocks (§5) are shared by NLB and GWLB too — only the routing intelligence differs.

05 The four building blocks of an ALB

An ALB is assembled from four kinds of objects. They build on each other, so we introduce them in dependency order — each one only uses concepts already defined.

5a · Targets & target groups

A target is something that receives traffic and runs your app. A target group is a named set of targets the load balancer routes to, using a protocol and port you specify. You register targets into a target group.

A target group has one of three target types, fixed at creation:

  • instance — EC2 instances, by instance ID
  • ip — private IP addresses (containers, on-prem servers reachable over VPN/Direct Connect, etc.)
  • lambda — a single Lambda function

The target group is also where you define health-check settings (next) — health is configured per target group, not per load balancer.

5b · Health checks

Your ALB "periodically sends requests to its registered targets to test their status." Only healthy targets get traffic; a target that fails enough consecutive checks is taken out of service, and put back when it recovers. Key defaults to know:

  • Path — the URL the check requests, default /. Point it at a lightweight endpoint (e.g. /health).
  • Success code — default 200. The target must return this for the check to pass.
  • Thresholds — consecutive successes to mark healthy (default 5) and failures to mark unhealthy (default 2), probed on an interval (default 30s).
Fail-open

If every target in a group is unhealthy, the ALB "fails open" — it routes to all of them anyway, on the logic that a struggling target beats no target at all. So a green dashboard isn't proof of health; watch the metrics in §11.

5c · Listeners

A listener is "a process that checks for connection requests, using the protocol and port that you configure." An ALB needs at least one — without it, the load balancer can't receive traffic. ALB listeners speak HTTP or HTTPS, on any port (typically HTTP:80 and HTTPS:443).

5d · Listener rules

A listener doesn't route by itself — its rules do. When a request arrives, the listener evaluates its rules in priority order (lowest number first). Each rule has:

  • Conditions — what to match: path-pattern, host-header, http-request-method, source-ip, http-header, query-string.
  • An action — what to do: forward to a target group, redirect to a URL, or return a fixed-response.

Every listener has a mandatory default rule, evaluated last, that catches anything no other rule matched. This is how one ALB serves many apps: /api/* → the API target group, example.com vs admin.example.com → different groups, everything else → the default.

06 How a request flows

Now the four blocks connect into one path. A client request travels like this:

Client

Resolves the LB's DNS name to an IP (§10).

→

Listener

Receives it on its protocol + port, e.g. :443.

→

Rule

First matching rule (by priority) decides the action.

→

Target group

A forward action sends it here.

→

Healthy target

One is picked (round robin by default) and gets the request.

To restate the routing logic from AWS: the ALB "evaluates the listener rules in priority order to determine which rule to apply, and then selects a target from the target group for the rule action." The default routing algorithm within a target group is round robin. The request reaches the target over its private IP address — targets never need a public IP.

07 Subnets, Availability Zones & scheme

To actually deploy an ALB you place it in your network. Recall Availability Zones from §2 — here they become a hard requirement.

At least two Availability Zones

You attach the ALB to subnets, and AWS creates a load balancer node in each zone you enable. For an ALB, "it is a requirement that you enable at least two or more Availability Zones." If one AZ goes dark, the ALB keeps serving from the others. (Practical note: give each subnet enough free IPs — AWS recommends at least a /27.)

Cross-zone load balancing

Each zone's node can either spread traffic only across targets in its own zone, or across targets in all enabled zones. The latter is cross-zone load balancing, and for ALB it's on by default — so 10 targets split unevenly across 2 AZs still each get an even share.

Scheme: internet-facing vs internal

At creation you choose one, and it can't change later:

  • Internet-facing — the nodes get public IPs; the DNS name resolves publicly. Use it for anything the internet must reach.
  • Internal — the nodes have only private IPs; reachable only from within the VPC. Use it for internal/back-end tiers.

Either way, the load balancer reaches your targets over private IPs. A common multi-tier design: an internet-facing ALB in front of web servers, and an internal ALB in front of the app servers behind them.

Sources: How Elastic Load Balancing works · Application Load Balancers (subnets, scheme)

08 Security groups — the firewall

An ALB has a security group: a virtual firewall that "controls the traffic that is allowed to reach and leave the load balancer." You specify which ports and protocols are allowed inbound and outbound. The rules must permit:

  • Inbound on the listener port (e.g. 443 from the internet for a public app).
  • Outbound to your targets on both the listener port and the health-check port (recall §5b — health checks are real connections too).
Best-practice pattern

Lock your targets' security group so it only accepts traffic from the load balancer's security group (use the LB's SG as the source). Now nothing can hit your servers except through the ALB — exactly the "single front door" from §1, enforced.

09 HTTPS & TLS termination

To serve HTTPS, you add a listener (§5c) on port 443 with the HTTPS protocol. That listener needs an SSL/TLS certificate, which you get for free from AWS Certificate Manager (ACM) and attach to the listener.

The ALB then performs TLS termination: it "uses a server certificate to terminate the front-end connection and then decrypt requests from clients before sending them to the targets." In other words, the encryption work happens at the load balancer, and your application servers receive plain HTTP — they don't manage certificates at all.

The standard setup uses two listeners, leaning on the rule actions from §5d:

  • HTTPS:443 — terminates TLS and forwards to your target group.
  • HTTP:80 — a single redirect rule bouncing everything to https://, so no one is served over plain HTTP.

Source: Create an HTTPS listener — AWS Docs

10 Reaching your app: the DNS name

This is the endpoint clients in §6 resolve. Every ALB gets an AWS-assigned DNS name of the form:

load balancer DNS name
my-load-balancer-1234567890abcdef.elb.eu-south-1.amazonaws.com

That name resolves to the ALB nodes' IP addresses — one per enabled AZ. Those IPs change over time as AWS scales the load balancer (the DNS record has a 60-second TTL), so you must never hard-code the IPs. Always reference the load balancer by its DNS name.

For a friendly address like app.example.com, map your own domain onto the ALB's DNS name. With Route 53 (AWS's DNS service) you create an alias record pointing at the load balancer; clients hit your domain, DNS resolves it to the ALB.

Sources: Application Load Balancers (DNS name) · How Elastic Load Balancing works (request routing)

11 Operating it: health & metrics

Once traffic flows, ELB publishes metrics to Amazon CloudWatch under the AWS/ApplicationELB namespace. The handful worth an alarm:

MetricWatch it for
HealthyHostCount / UnHealthyHostCountAre your targets passing health checks? Alarm on a non-zero UnHealthyHostCount.
RequestCountTraffic volume reaching targets.
TargetResponseTimeHow long your app takes to respond (latency). Track percentiles, not just the average.
HTTPCode_Target_5XX_CountServer errors coming from your app.
HTTPCode_ELB_5XX_CountErrors from the load balancer itself (e.g. no healthy target) — distinct from your app's errors.

For per-request detail (client IP, path, latencies, response codes), enable access logs, which the ALB writes to an S3 bucket you specify.

Connection draining

When you remove a target, the ALB doesn't cut its live connections. The target enters a draining state and finishes in-flight requests before going away — controlled by the target group's deregistration_delay.timeout_seconds (default 300s). This is what makes rolling deploys graceful.

Sources: CloudWatch metrics for your ALB · Target groups (deregistration delay)

12 Good habits & gotchas

  • Always span ≥2 AZs, with a healthy target in each. One target per zone is the minimum for the ALB to be effective.
  • Give health checks a real endpoint. A lightweight /health path returning 200 beats the default /, which might be heavy or redirect.
  • Least-privilege security groups. Targets accept traffic only from the ALB's security group (§8).
  • Terminate TLS at the ALB with an ACM certificate, and redirect HTTP→HTTPS (§9).
  • Front it with a Route 53 alias, never the raw node IPs (§10) — they change.
  • Pick the right type. ALB for HTTP/HTTPS; reach for NLB only when you need raw TCP/UDP, a static IP, or extreme performance (§4).
